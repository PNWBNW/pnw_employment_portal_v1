# Tech Stack — PNW Employment Portal

> All decisions here are locked. Change only with an explicit decision log entry.

---

## Framework: Next.js 15 (App Router)

**Why Next.js 15:**
- App Router enables layout-level auth guards (`(employer)/layout.tsx` wraps all
  employer routes with session check) with no extra setup
- React Server Components let us keep heavy computation (manifest compilation,
  hash derivation) in server-side code paths during SSR, reducing client bundle
- Static export (`output: 'export'`) lets us ship as a plain static site for
  Codespace/local testing without a server — same code, zero backend
- Excellent TypeScript support out of the box
- Compatible with Tauri (Rust desktop wrapper) for future desktop packaging —
  just point Tauri's webview at `localhost:3000` or the static export

**What we are NOT using:**
- Next.js API routes / server actions for any sensitive data. No server ever sees
  private keys, view keys, or wage data. The app is treated as client-first.
- Pages Router (legacy). App Router only.

---

## Language: TypeScript (strict)

```json
{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "exactOptionalPropertyTypes": true
  }
}
```

`any` is banned except at the adapter boundary (`src/lib/pnw-adapter/`) where
Aleo record shapes are inherently dynamic. Every `any` must have a comment.

---

## Styling: Tailwind CSS 4 + shadcn/ui

**Why Tailwind:**
- Utility-first makes the payroll table, status badges, and dense data UIs fast
  to build without a stylesheet explosion
- Consistent spacing and typography tokens

**Why shadcn/ui:**
- Radix UI primitives underneath — accessible by default
- Components are copied into the repo (`components/ui/`), not imported from npm
  This means we own the components and can modify them without version conflicts
- Pre-built: Table, Dialog, Tabs, Badge, Card, Input, Select, Toast, Sheet, Form
- Data table pattern (`TanStack Table` + shadcn DataTable) is exactly what the
  payroll table needs

**Component philosophy:**
shadcn/ui components live in `components/ui/` (generated by the CLI, do not edit).
Custom portal components live in `components/payroll-table/`, `components/run-status/`,
etc. Never mix the two directories.

---

## Data Table: TanStack Table v8

Used exclusively for the payroll table UI (`components/payroll-table/`).

**Why TanStack Table:**
- Headless (no opinionated styles) — we supply shadcn/ui table markup
- Handles 25+ rows with inline editing, column sorting, aggregation, and validation
- Row selection, inline cell editing, and controlled state via React are all built in
- Works with Zustand for shared state outside the table

---

## State Management: Zustand

Used for:
- `usePayrollRunStore` — PayrollRunManifest lifecycle (draft → anchored state machine)
- `useAleoSessionStore` — session keys (address, view_key; private_key never in Zustand)
- `useWorkerStore` — cached decoded worker records for the session

**Why Zustand:**
- Minimal boilerplate vs. Redux
- Works great with Next.js App Router (can be used in client components)
- Devtools available for debugging state machine transitions
- Easy to subscribe to specific slices without re-rendering everything

**What NOT to store in Zustand:**
- Private keys (session only, never global store — only in `sessionStorage`)
- Raw Aleo record ciphertext (too large; keep in a local cache with TTL)

## Hashing / Crypto: @noble/hashes (BLAKE3)

The same library used by `pnw_mvp_v2/portal/src/commitments/hash.ts`.

```
pnpm add @noble/hashes
```

All hash computation in this portal must use the same domain-separated BLAKE3
as `pnw_mvp_v2`. The manifest compiler imports `hash.ts` and `canonical_encoder.ts`
directly from `src/lib/pnw-adapter/` — the copied versions from pnw_mvp_v2.

**Do not use:**
- `crypto.subtle` SHA-256 for manifest hashing (wrong algorithm)
- Any other BLAKE3 library (different output)

## Testing: Vitest

```
pnpm add -D vitest @vitest/ui
```

**Test targets:**
- `src/manifest/compiler.test.ts` — row hashing, batch_id determinism
- `src/manifest/chunk_planner.test.ts` — chunking rules, edge cases (1 row, 25 rows)
- `src/coordinator/receipt_reconciler.test.ts` — payroll_inputs_hash matching
- `src/anchor/batch_anchor_finalizer.test.ts` — NFT arg construction
- `src/lib/pnw-adapter/*.test.ts` — copied from pnw_mvp_v2 adapter tests

**What we do not test with Vitest:**
- Next.js routing and page rendering (Playwright for E2E, added post-MVP)
- Actual snarkos execution (integration test, requires live testnet)

## PDF Generation: jspdf

Client-side only. Generates paystub, credential certificate, payroll run summary,
and audit authorization PDFs directly in the browser. No server upload, no
third-party service.

```
pnpm add jspdf
```

PDF components live in `components/pdf/`. They receive only display-safe data
(decoded amounts, user-chosen display labels, tx IDs, anchor hashes). They never
receive raw addresses, private keys, or view keys. Shared layout helpers are in
`components/pdf/pdf_helpers.ts`.

---

## QR Code: qrcode (+ react-qr-code for rendering)

Used for onboarding QR codes (employer generates → worker scans).

```
pnpm add qrcode react-qr-code
pnpm add -D @types/qrcode
```

QR payload schema is defined in `src/onboarding/qr_payload.ts`. Contents:
- Portal URL with onboarding route
- Pre-computed employer identifier hash (not the raw address)
- Agreement template ID
- Schema version

The QR never encodes a raw Aleo address or private key.

## Wallet Adapters: @provablehq/aleo-wallet-adaptor-*

The official Provable wallet adapter stack for Aleo.

```
pnpm add @provablehq/aleo-wallet-adaptor-react @provablehq/aleo-wallet-adaptor-react-ui
pnpm add @provablehq/aleo-wallet-adaptor-shield @provablehq/aleo-wallet-adaptor-puzzle
pnpm add @provablehq/aleo-wallet-adaptor-leo @provablehq/aleo-wallet-adaptor-fox
pnpm add @provablehq/aleo-wallet-adaptor-soter
pnpm add @provablehq/aleo-wallet-adaptor-core @provablehq/aleo-wallet-standard
pnpm add @provablehq/aleo-types
```

All packages at `0.3.0-alpha.3`. Follows the exact pattern from
[aleo-dev-toolkit-react-app.vercel.app/wallet](https://aleo-dev-toolkit-react-app.vercel.app/wallet).

**What each package does:**
- `aleo-wallet-adaptor-react` — `AleoWalletProvider` context + `useWallet()` hook
- `aleo-wallet-adaptor-react-ui` — `WalletModalProvider` + `WalletMultiButton` pre-built UI
- `aleo-wallet-adaptor-shield/puzzle/leo/fox/soter` — per-wallet adapters
- `aleo-wallet-adaptor-core` — `BaseAleoWalletAdapter`, error types, utilities
- `aleo-wallet-standard` — `WalletReadyState`, `WalletDecryptPermission` enums
- `aleo-types` — `Network` enum

**Mobile support:**
- Leo and Fox adapters accept `{ isMobile: true, mobileWebviewUrl: <dapp-url> }`
  config to redirect to their in-app browser (`app.leo.app/browser?url=...`)
- A custom `WalletMobileRedirectHandler` component handles the LOADABLE → deep-link
  fallback for wallets that don't support in-app browser config (Shield, Puzzle)

## Aleo SDK (Client-Side Record Decoding)

For decoding the employer's own private records (paystub receipts, USDCx balances)
from the Aleo network using their view key.

**Package:** `@provablehq/sdk` (browser WASM build — loaded dynamically)

View key scanning runs in a Web Worker to avoid blocking the main thread.

**Scope:** Used only for:
1. Decoding employer's own `WorkerPaystubReceipt` and `EmployerPaystubReceipt` records
2. Scanning for USDCx Token records owned by the employer
3. Post-MVP: worker side record decoding

**Never used for:** signing transactions. Signing goes through the wallet adapter
(`useWallet().signMessage`) or through the CLI adapter for on-chain execution.

## What We Are NOT Using

| Technology | Reason |
|-----------|--------|
| tRPC / GraphQL | No server; all data is from Aleo RPC + session |
| Prisma / Postgres | No database; no persistent storage of private data |
| NextAuth / Clerk | No traditional auth; session is Aleo wallet connection |
| Redux | Zustand is sufficient; Redux is too much boilerplate |
| Styled Components / Emotion | Tailwind is sufficient; CSS-in-JS adds bundle weight |
| React Query | Direct fetch from Aleo REST API is simple enough for MVP |
| @demox-labs/aleo-wallet-adapter-* | Replaced by official @provablehq/aleo-wallet-adaptor-* |
| @react-pdf/renderer | Replaced by jspdf (lighter, simpler API) |
| WalletConnect | Aleo wallets use in-app browser pattern, not WalletConnect |
